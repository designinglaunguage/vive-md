## 1. 변경 이력 작성 기준

### 1.1 기록 대상

| 구분 | 기록 여부 | 설명 |
|------|-----------|------|
| 결함 수정 | 필수 | 운영 중 발견된 버그, 장애, 데이터 오류 수정 |
| 기능 개선 | 필수 | 기존 기능의 UX, 성능, 정책, 동작 방식 개선 |
| 신규 기능 | 필수 | 운영 이후 추가된 기능 |
| 환경 변경 | 필수 | OS, 브라우저, 외부 API, 인프라, 의존성 변경 대응 |
| 보안 패치 | 필수 | 취약점 수정, 권한/인증/암호화 정책 변경 |
| 설정 변경 | 조건부 | 사용자 영향 또는 운영 영향이 있는 설정 변경 |
| 문서 변경 | 조건부 | 운영 절차, 정책, 사용자 안내에 영향을 주는 문서 변경 |

### 1.2 변경 유형

| 유형 | 의미 | 예시 |
|------|------|------|
| 수정 | 결함을 바로잡는 변경 | 로그인 오류 수정, 데이터 저장 버그 수정 |
| 적응 | 외부 환경 변화에 대응하는 변경 | 외부 API 버전 변경, 브라우저 호환성 대응 |
| 개선 | 사용자 경험이나 품질을 높이는 변경 | 검색 속도 개선, 화면 문구 개선 |
| 예방 | 장애나 보안 문제를 사전에 줄이는 변경 | 의존성 업데이트, 리팩토링, 모니터링 추가 |
| 보안 | 보안 위험을 줄이는 변경 | 권한 검증 강화, 토큰 만료 정책 변경 |
| 문서 | 운영/사용/개발 문서 변경 | 운영 가이드 수정, 장애 대응 절차 추가 |

### 1.3 영향도 기준

| 영향도 | 기준 | 승인 기준 |
|--------|------|-----------|
| 긴급 | 서비스 중단, 데이터 손상, 보안 사고 가능성 | 즉시 승인 후 사후 보고 |
| 높음 | 주요 기능 장애 또는 다수 사용자 영향 | 책임자 승인 필요 |
| 보통 | 일부 기능 또는 제한된 사용자 영향 | 팀 리드 승인 |
| 낮음 | 문구, 문서, 내부 개선 수준 | 담당자 검토 후 반영 |
